Brokered Tool Isolation One invocation, one micro sandbox

Each tool invocation runs in its own dedicated isolated micro sandbox.

The agent never talks directly to the isolated tool. All command requests, credentials, stdio, approvals, and audit events pass through the supervisor boundary.

A tool invocation gets a fresh ephemeral sandbox - scoped to one command policy decision.
The child receives only the capabilities, env vars, credentials, and network routes granted for that invocation.
When the invocation exits, its micro sandbox and transient broker state are discarded.
Tools can be chained (git can call ssh, but the agent cannot call ssh directly).

Policy follows the real command, even when an agent tries to hide it.

Brokered Tool Isolation turns command execution into a supervised policy decision at the moment a process tries to exec a tool.

Top-level and nested executions are mediated.
Rules match the resolved command and argv, not just the original shell line.
Allowed commands receive a scoped sandbox for that invocation.

The showcase is intentionally layered: argv policy blocks obvious high-level mutations, while L7 policy handles broad API tools where argv alone is too coarse.

Wrapping in shell or Python does not bypass nested command mediation.
Direct API writes are denied even when the command itself is allowed.
Credentials stay brokered throughout the request path.
