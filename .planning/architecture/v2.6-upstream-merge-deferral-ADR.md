## Context

- **Rebase scope exceeded runbook assumptions.** The 2026-04-29 rebase attempt on `origin/v2.0-pr` revealed 504 upstream commits between the old merge-base (`063ebad6`) and the current upstream tip (`34725154`), absorbing 4 minor releases (v0.41 → v0.42 → v0.43 → v0.43.1 → v0.44). The resulting conflict set spanned 77 files: 49 `UU` content conflicts, 26 `AA` add/add collisions, and 2 `DU` delete/modified pairs. Critically, 16 of the 26 `AA` collisions are `crates/nono-cli/src/*_runtime.rs` files — both upstream and v2.0-pr added these runtime files independently with architecturally divergent content. These are parallel-evolution collisions, not drift, and require architectural review per-file rather than textual merge. This scope exceeded the "drift-only" assumption embedded in the `260428-rsu-PLAN.md` 5-task runbook.
- **Upstream PRs 725 and 726 remain stalled.** PRs 725 (`feat(windows): windows gap closure (v2.0 milestone)`) and 726 (`feat(windows): resource limits + extended ipc + attach-streaming + cleanup (v2.1)`) are both OPEN + CONFLICTING + REVIEW_REQUIRED at Phase 46 plan-open. The 2026-04-29 outreach comments (issuecomment-4345050113 on PR 725; issuecomment-4345050118 on PR 726) offered three paths and asked the maintainer to pick one; no response has been received. PR 583 (the predecessor "refresh" PR) is CLOSED unmerged.
- **Three-milestone re-deferral history.** REQ-MERGE-01 has been explicitly re-deferred at v2.3 (2026-04-29 per `quick-260428-rsu`, commit `7911ef0e`), v2.4, and v2.5 scope-locks. The project's REQUIREMENTS.md (line 42) and PROJECT.md (lines 26 + 196) document this history. Each scope-lock concluded that the external gate (maintainer engagement) had not moved and that blocking the milestone on it was higher-cost than deferral.
- **ROADMAP v2.6 SC#1 provides an explicit alternative path.** Phase 46 ROADMAP Success Criterion 1 explicitly states: "`windows-squash` → `main` merge gate-state explicitly documented via feature-flag-equivalent rollout." This alternative path was designed at v2.6 planning time precisely because the pattern of indefinite-external-gate blocking was established by three prior scope-locks.
- **Per-phase umbrella PR pattern already in active use.** Phases 22, 33, 39, 42, and 43 all used the per-phase umbrella PR pattern to contribute work upstream without depending on the windows-squash merge gate. PR 922 (Phase 40) is the live exemplar. This pattern is captured in memory `project_cross_fork_pr_pattern` but has not previously been codified in a durable planning artifact.

## Goals

- Close REQ-MERGE-01 without blocking the v2.6 milestone close on an indefinite external-response gate, while preserving full revival optionality on maintainer engagement.
- Land an auditable ADR that makes the deferral disposition, the reasoning, and the go-forward mode discoverable to future phases without requiring re-derivation from scattered planning artifacts.
- Codify the per-phase umbrella PR pattern as the fork's documented go-forward upstream-contribution mode, replacing implicit-from-precedent with explicit-from-ADR.
- Maintain forward compatibility with a v3.0 re-evaluation of the upstream merge, so the decision made here does not foreclose future rebase or consolidation paths.

## Non-goals

- This ADR does NOT close the upstream merge as "won't do." PRs 725 and 726 remain OPEN; the 2026-04-29 outreach remains the canonical communication. The fork's intention to merge is unchanged; only the timeline gate is being replaced.
- This ADR does NOT abandon the 260428-rsu runbook. The 5-task force-rebase runbook at `260428-rsu-PLAN.md` remains a referenced abandoned-path artifact. On maintainer-response trigger, the runbook is a valid starting point (with scope recalibration for any additional drift accumulated since 2026-04-29).
- This ADR does NOT introduce a fork-side calendar trigger for the upstream merge. The rejected options included both a v3.0-milestone trigger and a drift-quantification threshold trigger; both were explicitly considered and rejected at D-46-A3. No Phase 46 artifact sets a calendar-based revival gate.
- This ADR does NOT replace the per-phase umbrella PR pattern with a "permanent fork" decision. The fork's long-term goal remains bidirectional parity with the always-further/nono upstream per the Phase 33 `continue` ADR option. D-46-A4 codifies the umbrella PR pattern as the contribution mode while PRs 725/726 are held — not as a permanent divergence commitment.

## Decision

Per ROADMAP Phase 46 Success Criterion 1 (SC#1), REQ-MERGE-01 is closed via the explicit alternative path: "feature-flag-equivalent rollout with the gate-state explicitly documented." This corresponds to Option A in the Decision Table above (D-46-A1).

The chosen path closes REQ-MERGE-01 without further rebase work by replacing the indefinite external-response gate with an auditable ADR that documents the deferral disposition. The fork's Windows implementation ships fully functional and security-equivalent to the Linux/macOS platforms; the `windows-squash` → `main` merge is a consolidation event, not a safety gate. This distinction justifies closing the requirement via the SC#1 alternative path.

This decision is reversible. The revival trigger set in the Consequences section below (D-46-A3) defines the specific maintainer-response signals that would reactivate the 260428-rsu runbook or a successor approach. The per-phase umbrella PR pattern codified in the Go-forward section (D-46-A4) continues to carry fork contributions to upstream while PRs 725/726 are held.

## Consequences

### Positive

- **v2.6 milestone close unblocked.** REQ-MERGE-01 no longer gates Phase 46 close on an external response that has not arrived in three prior milestones.
- **Per-phase umbrella PR pattern becomes auditable.** D-46-A4 promotes the pattern from implicit-in-memory to explicit-in-ADR; future phases can cite this document rather than reconstructing the precedent from Phases 22/33/39/42/43.
- **Revival optionality fully preserved.** PRs 725 and 726 remain OPEN; the 2026-04-29 outreach remains the canonical communication; the 260428-rsu runbook remains a referenced artifact. No path is foreclosed.
- **No fork-side calendar dependency.** Phase 47 (UPST6 audit) and Phase 48 (UPST6 sync) do not need to wait for the upstream merge to execute their scoped work.
- **Consistent with v2.5 close's already-chosen deferral on the same evidence.** This ADR does not change the disposition that v2.5 already adopted; it formalizes it with the explicit SC#1 alternative path and adds the per-phase umbrella PR codification.

### Negative

- **`windows-squash` and `main` continue to diverge.** Each upstream release ingested via Phases 47/48 adds to the delta. The drift cost that was 504 commits in April 2026 will be larger at any future revival trigger, especially given the 16 architectural AA collision files that require per-file review.
- **Future upstream sync cost may grow.** Phase 47 audit may quantify this growth; if the AA cluster drift accelerates, the revival scope at the next maintainer-response trigger will be larger than the April 2026 estimate.
- **Fork contributors see two branches without a consolidation signal.** Developers landing on the fork repository see `windows-squash` and `main` coexisting without a merge in-flight, which may be confusing without this ADR as a reference.
- **Maintainer-non-response state could persist indefinitely.** The 2026-04-29 outreach is the only active communication; no fresh outreach is being added by Phase 46 (D-46-A1). The window for receiving a response before v3.0 milestone start depends entirely on maintainer availability.
- **Revival trigger requires watchfulness on PR 725/726 activity.** No automated monitoring is added; manual polling via `gh pr view 725/726 --repo always-further/nono` remains the detection mechanism per the 260428-rsu Watch-list section.

### Revival triggers (maintainer-response only)

Resume scope determined at trigger time. ANY of:

1. Maintainer comments on PR 725 or 726 with directional guidance (rebase, close-and-restage, alternate approach).
2. Maintainer takes substantive action on either PR (review submitted, label change, close, merge).
3. Maintainer requests a different approach via issue, discussion, or direct communication.
4. (Explicitly NOT a trigger: v3.0 milestone calendar, drift-quantification threshold — per D-46-A3 these were considered and rejected. Fork-side calendar triggers are not set; no drift level automatically re-opens REQ-MERGE-01.)

### Go-forward upstream-contribution mode (per-phase umbrella PR)

While PRs 725/726 remain held, the fork's upstream contribution mode is the per-phase umbrella PR pattern (memory `project_cross_fork_pr_pattern`). Precedent: Phase 22 (UPST2), Phase 33 (this ADR's parent phase), Phase 39, Phase 42, Phase 43 (umbrella PR opened by Phase 46 Plan 46-02). PR 922 (Phase 40) is the live exemplar. GitHub's one-PR-per-branch-pair rule means per-plan upstream PRs require per-plan feature branches; the umbrella pattern collapses N plans into 1 PR per phase. This decision is codified here per D-46-A4 so that future phases can cite this ADR as the authoritative source rather than reconstructing the precedent from memory entries and commit history.

## Alternatives Considered

### Option B — Re-poll maintainer + decide on response

This option would have involved drafting a new outreach message to the upstream maintainer on PRs 725 and/or 726, potentially including a partial rebase of the v2.0-pr stack to demonstrate continued good faith, and then deferring the REQ-MERGE-01 close decision until the maintainer responded. The appeal is that it keeps the merge path active and avoids formalizing the deferral as a closed state.

The option was rejected for three reasons. First, the same reasoning — external gate has not moved, blocking is higher-cost than deferral — produced three prior scope-lock deferrals at v2.3, v2.4, and v2.5; a new outreach message does not change the fundamental asymmetry between the fork's contribution capacity and the maintainer's availability. Second, the timeline risk is indefinite: if the new outreach also receives no response, v2.6 would face a fourth scope-lock deferral without the SC#1 alternative path. Third, the 2026-04-29 outreach already offered three concrete paths and asked the maintainer to choose; no additional information from the fork's side would improve on that offer. The "re-poll" option was rejected at D-46-A1 explicitly in favor of accepting the external gate's current state and closing via the SC#1 path.

### Option C — Resume 260428-rsu force-rebase

This option would have executed the 5-task runbook at `260428-rsu-PLAN.md`: (1) rebase `v2.0-pr` onto upstream/main, (2) rebase `v2.1-pr` onto the result, (3) smoke-test both branches, (4) force-push to origin, and (5) clean up staging branches. The runbook was authored in April 2026 with the assumption that the conflict scope would be "drift-only" — mechanically resolvable textual conflicts from parallel evolution of the same code.

The empirical evidence from the 2026-04-29 rebase attempt disqualifies this path for the current scope. The 504 upstream commits and 77 conflict files were already beyond the runbook's size assumption, but the disqualifying factor is the 16 `AA` add/add collisions in `crates/nono-cli/src/*_runtime.rs`. Unlike the 49 `UU` content conflicts (which could in principle be mechanically resolved file-by-file), the AA cluster represents parallel architectural evolution: both upstream and v2.0-pr independently created these runtime files with substantively different implementations. The common ancestor (`063ebad6`) does not contain any of these files; there is no textual base from which to derive a merge result. Each of the 16 files requires an architectural review comparing the upstream design to the fork design, followed by a deliberate synthesis decision — a process that is bounded by contributor capacity and carries meaningful regression risk in the process. No maintainer engagement exists to validate that the synthesis decisions are acceptable upstream, which means the force-push at step (4) would be submitting architecturally synthesized code that the maintainer has not signaled interest in reviewing. The option was rejected at D-46-A1 on these grounds.
