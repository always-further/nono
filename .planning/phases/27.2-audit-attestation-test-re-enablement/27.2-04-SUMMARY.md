**1. [Rule 4 - Architectural] tracing WARN output goes to stdout, contaminating `audit verify --json` parser**

- **Found during:** Task 4 (human-verify) — running `cargo test -p nono-cli --test audit_attestation -- --nocapture` on the user's Windows host (the same Windows host where this plan executed).
- **Failure mode:** Both Test 1 and Test 2 panic with `parse verify json: Error("expected value", line: 1, column: 1)` on the `serde_json::from_slice(&verify_output.stdout).expect("parse verify json")` line (lines 444 and 497 post-edit, respectively).
- **Captured stdout for Test 2's `audit verify --json` invocation** (882 bytes; tracing-WARN prefix prepended to JSON):
  ```
  [2m2026-05-09T17:10:42.235613Z[0m [33m WARN[0m NONO_TEST_HOME override active: C:\Users\OMack\Nono\crates\nono-cli\target\test-artifacts\nono-audit-attestation-it-SHgdxC\home
  {
  ```

- **Captured stderr:** empty (`len=0`).
- **Root cause:** `crates/nono-cli/src/cli_bootstrap.rs::init_tracing()` builds the tracing subscriber with `tracing_subscriber::fmt()` and no `.with_writer()` call — so the subscriber defaults to `io::stdout()` (per the `tracing-subscriber` crate default). Phase 27.1 Plan 01 added `crate::config::warn_once_test_home(&path)` which fires `tracing::warn!("NONO_TEST_HOME override active: {}", path.display());` once per process when the seam env var is set. The integration test sets `NONO_TEST_HOME` on every `nono` invocation it makes (line 25 of `audit_attestation.rs`, locked at Phase 27.1), so the warning fires on every `audit verify --json` subprocess and pollutes the JSON output.
- **Why this surfaced now:** Phase 27.1's `#[ignore]` attributes masked the issue at Phase 27.1 close. This Plan 27.2-04 removes the `#[ignore]`s; the tests now run end-to-end and surface the bug.
- **Why NOT auto-fixed:** Phase 27.2 plan explicitly locks `crates/nono-cli/src/` byte-identical (D-27.2-12). Fixing the tracing-stdout pollution requires either (a) `cli_bootstrap.rs::init_tracing()` rerouting to stderr, (b) `audit_commands.rs::cmd_verify` suppressing tracing when `--json` is set, (c) a global `--silent` carry-through to subprocesses, or (d) test-harness regex-stripping of tracing prefixes before JSON parse. Each path has different blast radius and is a Rule 4 architectural decision exceeding Plan 27.2-04's test-only scope.
- **Proposed routing options for orchestrator:**
  1. **Reroute tracing to stderr globally** (recommended for both correctness and convention — most CLI tools route logs to stderr; see `tracing_subscriber::fmt::Layer::with_writer(std::io::stderr)`). Single-line fix in `init_tracing()`. Risk: changes ALL `nono` log routing, may surprise users tailing stdout for human-readable warnings; mitigated by the fact that tracing logs are explicitly diagnostic, not user-facing output.
  2. **Suppress tracing when `--json` is set** in commands that expose `--json` (audit verify, audit show, etc.). Targeted fix; preserves stdout tracing for human runs. Risk: adds `--json` awareness to bootstrap / per-command init.
  3. **Strip tracing prefix in tests via test-harness regex** before JSON parse. Test-only fix; production behavior unchanged. Risk: shifts the burden onto every future `--json`-asserting test.
  4. **Fold into v2.5-FU-3** (new follow-up entry alongside FU-1 + FU-2). Defers test re-enablement to v2.5; tests stay un-`#[ignore]`'d but fail until v2.5 fix lands.
- **Suggested fix path:** Option 1 (reroute tracing to stderr globally) — single-line change in a separate Phase 27.2 plan (e.g., `27.2-05` or a `27.2-fix-01` follow-up), with regression tests confirming `audit verify --json` stdout is now pure JSON and that tracing-warn lines correctly land on stderr. Honors Phase 27.2's spirit (close REQ-AAHX-01..03 fully) without forcing this Plan 04 to absorb the architectural decision.
- **Files that would need to change** (out of scope for Plan 27.2-04):
  - `crates/nono-cli/src/cli_bootstrap.rs` (re-route subscriber to stderr)
  - `crates/nono-cli/src/cli.rs` if `--json` flag awareness is added per Option 2
- **NOT a regression in Plan 27.2-04's deliverable.** The plan's structural deliverables (`#[ignore]` removal, ScopedEnvVar guard, JSON-shape adaptation, comment-block prune) are all correct and committed. The bug is in production code Phase 27.2-04 was contractually NOT permitted to touch (D-27.2-12).

## Issues Encountered
