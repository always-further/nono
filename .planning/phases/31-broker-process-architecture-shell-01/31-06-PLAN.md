```mdx
## Windows nono shell — security envelope

`nono shell --profile claude-code --allow-cwd` on Windows 10/11 spawns a sandboxed PowerShell child via the broker-process architecture (Phase 31, validated 2026-05-XX). The token shape is:

- **`nono.exe`** runs at the user's identity (Medium IL or higher).
- **`nono-shell-broker.exe`** (sibling of `nono.exe`, Authenticode-signed with the same key) is spawned by `nono.exe` at the user's identity (Medium IL). It exists as an intermediary because directly spawning a Low-IL child from `nono.exe` triggers `STATUS_DLL_INIT_FAILED (0xC0000142)` at CSRSS console-subsystem attach time on Windows — the Low-IL ALPC client cannot reach the CSRSS console handler port, and KernelBase fails the DllMain.
- **`nono-shell-broker.exe`** duplicates its own token via `DuplicateTokenEx(SecurityAnonymous, TokenPrimary)` and lowers the duplicate to Low Mandatory Integrity Level via `SetTokenInformation(TokenIntegrityLevel, WinLowLabelSid)`.
- **The broker spawns the actual shell child** (PowerShell 5.1 / cmd.exe) via `CreateProcessAsUserW(low_il_token, ..., dwCreationFlags=EXTENDED_STARTUPINFO_PRESENT)`. NO `CREATE_NEW_CONSOLE`, NO `PROC_THREAD_ATTRIBUTE_PSEUDOCONSOLE` — the Low-IL child INHERITS the broker's already-attached console; KernelBase short-circuits the CSRSS attach because a console handle is already inherited.
- **The Low-IL shell child is the user-running shell**. Mandatory-label `NO_WRITE_UP` enforces write-deny via the MIC kernel pre-DACL check: a Low-IL subject (the shell process) cannot write to a Medium-IL object (paths outside the grant set, e.g., the user's Desktop), regardless of the file's DACL.

**What's enforced at OS level:**
- Filesystem write-deny outside the grant set (mandatory-label NO_WRITE_UP — kernel-level, NOT a hook check).
- Reads of granted paths still succeed (per-path Low-IL labels with read-pass mode masks via `try_set_mandatory_label`).
- Job Object containment caps the broker's process tree; the Low-IL grandchild inherits Job Object membership automatically because `JOB_OBJECT_LIMIT_*BREAKAWAY*` flags are unset.

**What's defense-in-depth (NOT the primary boundary):**
- The Claude Code `PreToolUse` hook intercepts attempted writes to disallowed paths and surfaces a prompt. If the hook fails (or is misconfigured, or is bypassed by an attacker controlling the agent's actions), the OS-level mandatory-label check still denies the write. The OS check is the security boundary; the hook is UX + audit.
- Network differentiation per-session via WFP `FWPM_CONDITION_ALE_USER_ID` is waived on this path (broker→Low-IL child); falls back to AppID-based filtering (same as Phase 15 detached-path waiver). For full per-session network filtering, use `nono run` (non-PTY supervised path retains WRITE_RESTRICTED + per-session SID).

**What's NOT enforced:**
- Low-IL does not block reads of Medium-IL files (NO_READ_UP is NOT set on the kernel default). For read-deny, use single-path grants with explicit Low-IL labels via Phase 21 `nono shell --read-only-grants` (carry-forward from Phase 21 design).
- Network deny within allow (Landlock-style); WFP allow-list is positive-only on Windows.

**Audit trail (Phase 28):** the Authenticode chain-walker (`AUDC-01..03`) records `signer_subject` + SHA-1 thumbprint for both `nono.exe` AND the broker process — both signed with the same key, so the audit trail records a consistent identity for the supervisor stack.
```

The exact prose is operator-/writer-discretion; the structural elements (token shape, what's enforced at OS level, what's defense-in-depth) are non-negotiable per CONTEXT Acceptance #6.
