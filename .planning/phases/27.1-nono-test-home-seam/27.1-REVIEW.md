# Phase 27.1: Code Review Report

**Reviewed:** 2026-05-04
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

Reviewed the Phase 27.1 NONO_TEST_HOME seam: a new `nono_home_dir()` helper in `crates/nono-cli/src/config/mod.rs` plus the migration of 15 home-dir resolution callsites across 11 source files (and one integration test). The seam is well-designed in isolation — env-var validation is correctly fail-closed on non-absolute values, `OnceLock`-guarded warn emission is bounded, and the migration preserves the original error-shape semantics on the standard `?`-propagation sites. The `clippy::unwrap_used -D warnings` bar is respected; no new `.unwrap()` / `.expect()` outside test code.

However, the migration also surfaced or carried forward several quality defects:

1. The fork now has TWO different home-dir resolvers — `validated_home()` (security-sensitive: HOME/USERPROFILE chain, ignores NONO_TEST_HOME) and `nono_home_dir()` (test-seam: honors NONO_TEST_HOME, calls `dirs::home_dir()` on miss). Several callsite pairs that conceptually refer to "the same home" now resolve to different directories under partial misconfiguration. This is a confused-deputy / split-brain hazard with security-policy-relevant consequences (audit logs and sensitive-path overlap rules can target distinct home directories within the same nono invocation).
2. The migration touched two `shorten_home` helpers that perform path-prefix matching via `String::strip_prefix` on `Path::display().to_string()` — the exact CLAUDE.md "Common Footguns #1" pattern. Pre-existing, but the edits effectively re-blessed it.
3. `user_state_dir()` deliberately returns `Some(<platform default>)` when NONO_TEST_HOME is non-absolute, while `nono_home_dir()` returns `Err(EnvVarValidation)` for the same input. This creates an asymmetric Windows surface where `rollback_root()` succeeds with an unexpected path and `audit_root()` fails — opposite of "fail secure on the same input."
4. The deferred Phase 27 integration test (`rollback_signed_session_verifies_from_audit_dir_bundle`) asserts a JSON shape that does not match production output. Currently `#[ignore]`'d so it doesn't fail CI, but the test as written cannot pass even after v2.4-FU-2 lands.

No correctness regressions in the basic migration mechanics were found. Findings below are in order of severity.

## Warnings

### WR-01: Split-brain home resolution between `validated_home()` and `nono_home_dir()` enables confused-deputy across security-policy boundaries

**File:** `crates/nono-cli/src/config/mod.rs:32-78` (validated_home), `crates/nono-cli/src/config/mod.rs:120-133` (nono_home_dir), `crates/nono-cli/src/policy.rs:229,232,234,467` and `crates/nono-cli/src/protected_paths.rs:43` (cross-callsite asymmetry)

**Issue:** Phase 27.1 deliberately keeps two home-dir resolvers:

- `validated_home()` reads HOME (Unix) or HOME → USERPROFILE → HOMEDRIVE+HOMEPATH (Windows). Used by `policy.rs::sensitive_paths()` (sensitive-path overlap rules), `policy.rs::expand_path` (capability resolution), and `learn.rs`/`query_ext.rs`/`sandbox_prepare.rs` for security-sensitive path evaluation.
- `nono_home_dir()` honors NONO_TEST_HOME first, then falls through to `dirs::home_dir()`. Used by `audit_root()`, `rollback_root()` (Unix arm), `protected_paths::from_defaults()` Unix arm, the `~`-expansion sites in `package_cmd.rs` / `trust_scan.rs`, and the `shorten_home` display sites.

Plan 01 SUMMARY (T-27.1-05 disposition) calls this separation "intentional" — security-sensitive code "continues to use validated_home() strict HOME-chain validation." But the two functions are NOT independent: the same conceptual `~/.nono` parent is resolved by both, just from different env-var sources. Concrete attack/misconfig scenario, on Unix:

```
HOME=/home/victim
NONO_TEST_HOME=/tmp/attacker
```

After this Phase:

- `policy::get_sensitive_paths()` expands `~/.ssh` → `/home/victim/.ssh` (validated_home → HOME).
- `audit_root()` writes session metadata to `/tmp/attacker/.nono/audit/` (nono_home_dir → NONO_TEST_HOME).
- `protected_paths::from_defaults()` Unix arm refuses grants overlapping `/tmp/attacker/.nono` — the *attacker-controlled* directory. The user's real `/home/victim/.nono` is no longer in the protected-roots list.
- `rollback_root()` Unix arm writes snapshots to `/tmp/attacker/.nono/rollbacks/`.

So a child of nono can have its audit trail and rollback snapshots silently written to attacker-controlled storage while sensitive-path overlap is still computed against `/home/victim`. Additionally, `protected_paths` no longer protects the real user `~/.nono` because its lookup also flipped to `nono_home_dir()`. This is the textbook confused-deputy pattern: half the security policy uses one principal's home, the other half uses a different principal's home.

CLAUDE.md § Path Handling explicitly says "Validate environment variables before use. Never assume HOME, TMPDIR, etc. are trustworthy." The `nono_home_dir()` helper validates only `is_absolute()`; it does NOT canonicalize, does NOT reject `..` segments inside an absolute path (e.g. `C:\Users\test\..\..\Windows\System32` passes `is_absolute()`), and does NOT cross-check that NONO_TEST_HOME and HOME refer to the same identity.

The Plan 01 SUMMARY (D-27.1-08) records: "the override is always-on in release binaries." That is the load-bearing decision behind this finding: NONO_TEST_HOME is not a debug-only seam; it is a production env-var honored by every shipped binary.

**Fix:** Choose one of:

1. **Recommended — close the gap between resolvers.** Make `nono_home_dir()` call `validated_home()` as its fallthrough (instead of `dirs::home_dir()`), so when NONO_TEST_HOME is unset both resolvers return the same path. When NONO_TEST_HOME IS set, additionally require the resolved path to be under, or equal to, the validated HOME chain — failing closed otherwise. This preserves the test-harness use case (CI sets HOME and NONO_TEST_HOME to the same tempdir) while killing the split-brain.

2. **Alternative — gate NONO_TEST_HOME on a build feature.** Move the env-var read behind `#[cfg(any(test, feature = "test-trust-overrides"))]` (or a new `test-home` feature), so production binaries simply do not honor it. This contradicts D-27.1-08 but eliminates the production EoP/confused-deputy surface entirely.

3. **Alternative — make `validated_home()` honor the same override.** Either both resolvers honor NONO_TEST_HOME or neither does. Splitting them is what produced this finding.

Either fix should also extend `nono_home_dir()` to canonicalize and reject `..` components, matching the CLAUDE.md path-handling stance.

### WR-02: `shorten_home` uses string-form `strip_prefix` on path data — CLAUDE.md Common-Footgun #1

**File:** `crates/nono-cli/src/audit_commands.rs:723-732`, `crates/nono-cli/src/rollback_commands.rs:243-252`

**Issue:** Both `shorten_home` helpers do this:

```rust
fn shorten_home(path: &Path) -> String {
    let s = path.display().to_string();
    if let Ok(home) = crate::config::nono_home_dir() {
        let home_str = home.display().to_string();
        if let Some(rest) = s.strip_prefix(&home_str) {
            return format!("~{rest}");
        }
    }
    s
}
```

This is the literal anti-pattern CLAUDE.md § Common Footguns #1 forbids: `&str::strip_prefix` on stringified paths. If `home = /home/user` and `path = /home/user2/secrets`, this produces `~2/secrets`. If `home = C:\Users\Tester` and `path = C:\Users\TesterEvil\file` on Windows (case-insensitive), the comparison is case-sensitive at the string level so it under-matches; if `home = C:\Users\Tester` and `path = C:\Users\Tester\Project`, it works — but `\Users\Tester2\...` would be silently mis-shortened to `~2\...`.

Phase 27.1 Plan 02 only changed the resolver call (`dirs::home_dir()` → `crate::config::nono_home_dir()`); the string comparison is pre-existing. However, the migration touched these lines, the helpers are display-only, and the fix is mechanical. Although display-only, mis-shortening can also feed into log forensics, support workflows, and the audit-list output (which groups by tracked path and shortens for readability — wrong shortening could mislead operators).

Note that `expand_home()` in `trust_scan.rs:481-488` and `expand_tilde()` in `package_cmd.rs:1010-1019` use `strip_prefix("~/")` which is a literal `~/` token check, NOT a path comparison — those are safe.

**Fix:** Use `Path::strip_prefix` (component-wise) instead of `String::strip_prefix`:

```rust
fn shorten_home(path: &Path) -> String {
    if let Ok(home) = crate::config::nono_home_dir() {
        if let Ok(rest) = path.strip_prefix(&home) {
            // rest is a relative Path; rejoin using OS separator, prepended with "~/"
            return if rest.as_os_str().is_empty() {
                "~".to_string()
            } else {
                format!("~{}{}", std::path::MAIN_SEPARATOR, rest.display())
            };
        }
    }
    path.display().to_string()
}
```

`Path::strip_prefix` does component-wise comparison and rejects `/home/user2/secrets` as not under `/home/user`. The existing `rollback_commands::shorten_home_replaces_prefix` test will need updates to accept the OS-separator-aware output (it currently asserts forward-slash on Unix, backslash on Windows — the proposed fix preserves that).

### WR-03: `user_state_dir()` and `nono_home_dir()` disagree on non-absolute NONO_TEST_HOME — Windows path asymmetry

**File:** `crates/nono-cli/src/config/mod.rs:120-133` (nono_home_dir), `crates/nono-cli/src/config/mod.rs:163-181` (user_state_dir), `crates/nono-cli/src/rollback_session.rs:30-43` (Windows rollback_root consumer)

**Issue:** When `NONO_TEST_HOME` is set to a non-absolute value (e.g. `relative/path`):

- `nono_home_dir()` returns `Err(NonoError::EnvVarValidation { var: "NONO_TEST_HOME", ... })` — fail-closed.
- `user_state_dir()` falls through to `dirs::state_dir().or_else(dirs::data_local_dir)` — returns `Some(<platform default>/nono)` and the override is silently ignored.

On Windows specifically, `rollback_root()` uses `user_state_dir()` exclusively:

```rust
#[cfg(target_os = "windows")]
{
    crate::config::user_state_dir()
        .ok_or(NonoError::ConfigParse(
            "Could not determine Windows state directory for rollback storage".to_string(),
        ))
        .map(|root| root.join("rollbacks"))
}
```

So on Windows with `NONO_TEST_HOME=relative/path`:

- `audit_root()` (via `nono_home_dir()`) errors with `EnvVarValidation`.
- `rollback_root()` (via `user_state_dir()`) succeeds and returns `%LOCALAPPDATA%\nono\rollbacks` — silently ignoring the (broken) override.

The same nono invocation gets two different states: the audit subsystem fails closed, the rollback subsystem silently uses the production directory the user did not intend. This violates the CLAUDE.md "fail secure" principle for the rollback path. The Plan 01 SUMMARY explicitly documents this fall-through as intentional — but it makes the helper-pair contract internally inconsistent and is the kind of detail that produces hard-to-debug Windows test flakes (the very class Phase 27 surfaced).

**Fix:** Make `user_state_dir()` symmetric with `nono_home_dir()`. Two options:

1. Treat non-absolute NONO_TEST_HOME as `None` (suppress, fail closed at the next `?`):
   ```rust
   pub fn user_state_dir() -> Option<PathBuf> {
       if let Ok(value) = std::env::var("NONO_TEST_HOME") {
           let path = PathBuf::from(&value);
           if !path.is_absolute() {
               return None; // fail-closed: caller's `?` will surface a SnapboxOrConfig error
           }
           return Some(path.join(".nono"));
       }
       dirs::state_dir().or_else(dirs::data_local_dir).map(|p| p.join("nono"))
   }
   ```
2. Or change `user_state_dir()` to return `Result<PathBuf>` and propagate `EnvVarValidation`. Larger blast radius (changes signature), but matches `nono_home_dir()` exactly.

Either way, document the intended consumer contract in the helper doc comment.

### WR-05: Integration test mutates process-wide environment via `std::env::set_var` / `remove_var` without RAII restore

**File:** `crates/nono-cli/tests/audit_attestation.rs:347-365`

**Issue:** The test does:

```rust
#[allow(clippy::disallowed_methods)]
{
    std::env::set_var(&env_var, &secret);
}
let keyref = format!("env://{env_var}");
// ... long block including subprocess invocation, assertions ...
let run_output = run_nono(&args, &home, &workspace);
#[allow(clippy::disallowed_methods)]
{
    std::env::remove_var(&env_var);
}
assert_success(&run_output);
```

Three problems:

1. **No RAII restore.** If `run_nono` panics (e.g. tempdir setup failed earlier in the test, or the assertion at line 366 fails for *any* reason), `remove_var` never executes and the env var leaks into other parallel tests. The unit-test `EnvVarGuard` pattern is exactly what this should use; the SUMMARY admits ENV_LOCK is `#[cfg(test)]`-gated to the unit-test compile unit and thus unavailable here, but a local RAII guard could be defined in the integration test file with ~10 lines.
2. **Cross-test races.** Cargo runs integration tests in parallel by default. `std::env::set_var` is documented as unsafe in newer Rust editions because libc `setenv` is not thread-safe with concurrent `getenv`. Even with PID+nanos suffix avoiding *name* collisions, the simultaneous mutation of the env table from multiple threads can corrupt internal libc state (especially on Linux glibc < 2.36). The `#[allow(clippy::disallowed_methods)]` annotation acknowledges this without mitigating it.
3. **The `assert_success(&run_output)` at line 366 fires AFTER the `remove_var`.** That's correct ordering for cleanup, but inverts the panic-safety story: assertion failure = clean (env restored) but tempdir setup failures earlier in the test (lines 322-345) panic with the env *not yet set*, while a panic in `run_nono` itself happens with the env set and not yet cleared. The leakage window is narrow but real.

The test is `#[ignore]`'d, so no CI impact today. When v2.4-FU-1 + v2.4-FU-2 land and the `#[ignore]` is removed, this leakage hazard becomes live.

**Fix:** Replace the explicit `set_var`/`remove_var` pair with a local RAII guard:

```rust
struct ScopedEnvVar { name: String }
impl ScopedEnvVar {
    fn set(name: impl Into<String>, value: &str) -> Self {
        let name = name.into();
        #[allow(clippy::disallowed_methods)]
        unsafe { std::env::set_var(&name, value); }
        Self { name }
    }
}
impl Drop for ScopedEnvVar {
    fn drop(&mut self) {
        #[allow(clippy::disallowed_methods)]
        unsafe { std::env::remove_var(&self.name); }
    }
}

let _key_env = ScopedEnvVar::set(&env_var, &secret);
let keyref = format!("env://{env_var}");
let run_output = run_nono(&args, &home, &workspace);
assert_success(&run_output);
// _key_env drops here, removing the var even on assertion-failure unwind.
```

This does not solve problem #2 (cross-test libc races) — the proper solution there is a process-wide test mutex, but per the SUMMARY ENV_LOCK is unavailable to integration tests. At minimum, document the race as a known hazard in the test source and consider running `audit_attestation.rs` integration tests serially via `cargo test -p nono-cli --test audit_attestation -- --test-threads=1`.

## Info

### IN-01: `nono_home_dir()` accepts paths containing `..` segments

**File:** `crates/nono-cli/src/config/mod.rs:120-133`

**Issue:** The validator only checks `Path::new(&value).is_absolute()`. Any of these are accepted:

- `C:\Users\test\..\..\..\Windows\System32` (Windows)
- `/home/user/../../etc` (Unix)

Both `is_absolute()` evaluate true. The path is then used to derive `<NONO_TEST_HOME>/.nono/audit`, `<NONO_TEST_HOME>/.nono/rollbacks`, etc. without further canonicalization at this layer. Downstream, `protected_paths::resolve_path` and `validate_path_within` do canonicalize, so the actual filesystem operations are safe; but the in-memory representation of "home" can mislead downstream string-comparison sites (see WR-02) and forensic logs (the `tracing::warn!` line emits the raw, un-canonicalized override).

**Fix:** After `is_absolute()`, also reject paths whose `components()` contain `Component::ParentDir`:

```rust
if path.components().any(|c| matches!(c, std::path::Component::ParentDir)) {
    return Err(NonoError::EnvVarValidation {
        var: "NONO_TEST_HOME".to_string(),
        reason: format!("must not contain '..' components, got: {}", value),
    });
}
```

This mirrors the input-string pre-check in `package_cmd::validate_relative_path` and matches CLAUDE.md § Path Handling.

### IN-02: `warn_once_test_home` race window allows duplicate emission

**File:** `crates/nono-cli/src/config/mod.rs:135-143`

**Issue:** Two threads can simultaneously read `WARNED.get().is_none() == true` before either calls `WARNED.set(())`. Both then emit the warn line; only one `set` succeeds, the other returns `Err(())` which is discarded. Comment at line 140 acknowledges "Worst case: warn fires twice across racing threads."

The bound is small (at most N threads on first contended access), so this is informational. Worth noting because Plan 01 SUMMARY claims "exactly one tracing::warn! per process" — the actual contract is "at-most-N-on-first-call, at-most-one steady state."

**Fix:** Use `OnceLock::get_or_init` to atomically gate the side effect:

```rust
fn warn_once_test_home(path: &Path) {
    static WARNED: std::sync::OnceLock<()> = std::sync::OnceLock::new();
    WARNED.get_or_init(|| {
        tracing::warn!("NONO_TEST_HOME override active: {}", path.display());
    });
}
```

This collapses the get/emit/set sequence into a single atomic init, so the warn fires exactly once per process across threads.

### IN-04: `legacy_windows_state_dir()` uses `validated_home()`, diverging from the rest of the Phase 27.1 unification

**File:** `crates/nono-cli/src/config/mod.rs:194-197`

**Issue:** `legacy_windows_state_dir()` (Windows-only) is consumed by `protected_paths::from_defaults()` Windows arm (line 31). It calls `validated_home()` rather than `nono_home_dir()`. Result on Windows with `NONO_TEST_HOME=C:\nono-test`:

- `protected_paths` includes `C:\nono-test\.nono` (from `user_state_dir()`).
- `protected_paths` ALSO includes `<USERPROFILE>\.nono` (from `legacy_windows_state_dir()` → `validated_home()` → USERPROFILE chain).

This is arguably correct (both legacy and current state dirs are protected, regardless of override) but it inherits the WR-01 split-brain: a test or production caller sets NONO_TEST_HOME and gets BOTH the override-rooted protection AND the original-home-rooted protection. If WR-01 is fixed by making `validated_home()` ALSO honor NONO_TEST_HOME, this will fold automatically. If WR-01 is fixed by removing NONO_TEST_HOME from production, this remains correct unchanged.

**Fix:** No standalone fix recommended — defer until WR-01 is resolved, then re-evaluate whether `legacy_windows_state_dir()` should switch to `nono_home_dir()`.
