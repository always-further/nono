  shape flipped from silent-default to fail-closed).

## Warnings

### WR-01: Test-isolation lock divergence from documented crate-wide invariant

**File:** `crates/nono-cli/src/trust_cmd.rs:2206`
**Issue:** The new tests introduce a local `OIDC_ENV_LOCK: Mutex<()>` for
serializing mutations of `NONO_TRUST_OIDC_ISSUER`. The same crate documents
at `crates/nono-cli/src/test_env.rs:1-10` (which the tests in this very file
use elsewhere, e.g. `trust_cmd.rs:1899` `let _lock = crate::test_env::lock_env();`)
that "All env-mutating tests must acquire this lock before touching env vars."
By adding a second, disjoint lock the change documents a precedent that future
contributors may follow: each new env-var gets its own private lock. The risk
is not in the present tests (the var-set is disjoint from any other env-mutating
test in the binary) but in the next reviewer/contributor who reads only one of
the two patterns and adds a test that reads `NONO_TRUST_OIDC_ISSUER` without
acquiring the right lock — re-introducing a race in CI.

The Phase 44.1 PLAN's stated rationale for a local mutex ("the test mutex must
be local — it cannot share `signing.rs`'s lock cross-crate") is correct for
the previous cross-crate situation, but does NOT justify a separate lock from
`crate::test_env::ENV_LOCK` since both the new helper and the helper-tests now
live in the same crate.

**Fix:** Replace the local `OIDC_ENV_LOCK` and `OidcEnvGuard::_lock` field
with the existing crate-wide lock. Either:

```rust
// Option A — reuse ENV_LOCK directly.
impl OidcEnvGuard {
    fn set(val: &str) -> Self {
        let _lock = crate::test_env::lock_env();
        let prev = std::env::var("NONO_TRUST_OIDC_ISSUER").ok();
        std::env::set_var("NONO_TRUST_OIDC_ISSUER", val);
        Self { _lock, prev }
    }
    // ...same for remove()
}
```

Or rebuild the new tests on top of `crate::test_env::EnvVarGuard::set_all`
(which already wraps the documented lock):

```rust
#[test]
fn read_required_oidc_issuer_returns_env_value_when_user_unset_and_env_set() {
    let _lock = crate::test_env::lock_env();
    let _env = crate::test_env::EnvVarGuard::set_all(&[
        ("NONO_TRUST_OIDC_ISSUER", "https://token.actions.githubusercontent.com"),
    ]);
    let got = read_required_oidc_issuer(None)
        .expect("explicitly-set env-var should succeed");
    assert_eq!(got, "https://token.actions.githubusercontent.com");
}
```

Either form keeps the testability properties prescribed by the PLAN while
preserving the documented crate-wide env-mutation lock invariant.

## Info

### IN-01: Asymmetric URL-shape validation between `--issuer` and `NONO_TRUST_OIDC_ISSUER`

**File:** `crates/nono-cli/src/trust_cmd.rs:937-960`
**Issue:** `read_required_oidc_issuer` eagerly URL-shape-validates the env-var
value via `Url::parse` (line 956), but accepts `user_issuer: Some(s)` and
returns `Ok(s.to_string())` without any shape check (line 940-942). Operators
who pass `--issuer ""` or `--issuer "not a url"` get a downstream error from
`validate_oidc_issuer` (`"OIDC issuer validation failed: ..."`) rather than
the helper's clear `"... is not a valid URL: ..."` message — and an operator
who passes `--issuer "   "` is NOT caught by the "whitespace-only treated as
unset" trap that the env-var path codifies (line 945). The fail-closed
contract still holds (downstream rejects), but the contract is asymmetric in
both error quality and the whitespace-trap coverage.

The plan-prescribed shape (PLAN lines 489-512) matches what was implemented,
so this is being filed as INFO rather than WARNING; the asymmetry is an
intentional simplification. Worth a follow-up to apply the same trim +
`Url::parse` gate to `user_issuer` for symmetry, OR to document explicitly
in the helper's docstring that `--issuer` shape validation is delegated to
the downstream `validate_oidc_issuer` step.

**Fix (optional):** Apply identical validation to both inputs:

```rust
fn read_required_oidc_issuer(user_issuer: Option<&str>) -> std::result::Result<String, String> {
    let candidate = match user_issuer.map(str::trim).filter(|s| !s.is_empty()) {
        Some(s) => s.to_string(),
        None => std::env::var("NONO_TRUST_OIDC_ISSUER")
            .ok()
            .filter(|v| !v.trim().is_empty())
            .ok_or_else(|| {
                "keyless bundle requires --issuer <OIDC_URL> or NONO_TRUST_OIDC_ISSUER \
                 env-var (exact match against signer's iss claim)"
                    .to_string()
            })?,
    };
    Url::parse(&candidate)
        .map_err(|e| format!("issuer '{candidate}' is not a valid URL: {e}"))?;
    Ok(candidate)
}
```

### IN-02: "Exact match" CLI doc wording is slightly imprecise

**File:** `crates/nono-cli/src/cli.rs:3047`
**Issue:** The doc reads "exact match against signer's iss claim", but the
downstream `trust::signing::validate_oidc_issuer` performs URL-component
equality (scheme + host + port), intentionally ignoring path/query/fragment
(see `signing.rs:53-56`). An operator reading the help text who passes
`--issuer https://gitlab.com/` (trailing slash) versus an `iss` claim of
`https://gitlab.com` may expect rejection (string-exact mismatch) but get
acceptance (URL-component equal). The semantics are arguably stronger than
"exact match", but the help-text phrasing creates a small expectation gap.

**Fix:** Tighten the wording slightly:

```rust
/// OIDC issuer URL (REQUIRED for keyless verify, unless NONO_TRUST_OIDC_ISSUER
/// is explicitly set; URL-component match (scheme + host + port) against
/// the signer's iss claim).
```

### IN-03: Comment block above `read_required_oidc_issuer` callsite duplicates 12 lines verbatim

**File:** `crates/nono-cli/src/trust_cmd.rs:1023-1035` and `crates/nono-cli/src/trust_cmd.rs:1218-1230`
**Issue:** The comment block describing the Phase 44.1 / T-44-01 / CR-01
fix is reproduced verbatim above both keyless verify callsites (12 lines
each). Total documentation duplication is acceptable for clarity at security
boundaries, but a future edit to one block (e.g., updating a citation) is
likely to drift from the other. The pattern would be cheaper to maintain
either as a module-level docstring above `read_required_oidc_issuer` itself
(it already has one — could be extended) with the callsite blocks
collapsed to `// Phase 44.1: see read_required_oidc_issuer docs.`

**Fix:** Collapse the two duplicate blocks to a single-line back-reference,
and consolidate the policy rationale into the helper's existing docstring
(`trust_cmd.rs:909-936`).
