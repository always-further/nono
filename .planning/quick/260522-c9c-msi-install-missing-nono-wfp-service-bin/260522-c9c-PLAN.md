<objective>
Fix the Windows MSI installers so that the machine-scope MSI bundles both
`nono-wfp-service.exe` and `nono-wfp-driver.sys` (the runtime currently fails at startup
because both files are absent under `INSTALLFOLDER`). User-scope MSI is explicitly held
to the three-binary set (`nono.exe`, `nono-shell-broker.exe`, docs) because a kernel
driver cannot run from per-user `LocalAppData` and the runtime already emits a
fail-closed error directing the user to the machine MSI.

Three observable wins:

1. POC user re-installs the next machine-scope MSI build → `[INSTALLFOLDER]\nono-wfp-service.exe`
   and `[INSTALLFOLDER]\nono-wfp-driver.sys` exist on disk; `nono setup --check-only` no
   longer reports `BackendBinaryMissing`.
2. CI's MSI emit-only step (added in an earlier phase) verifies the new
   `-DriverBinaryPath` plumbing so regressions are caught pre-release.
3. The currently-untracked `dist/windows/nono-machine.wxs` becomes a tracked reference
   manifest that matches what `build-windows-msi.ps1` actually emits, closing the
   "static .wxs vs generated .wxs drift" gap that contributed to this bug.

Purpose: Unblock POC users from `BackendBinaryMissing` startup failures on Windows
without breaking the existing machine-MSI service-registration lifecycle
(`<ServiceInstall>` + `<ServiceControl>` already work; we are only adding the driver
file harvest + a parallel reference manifest commit).

Output: Updated MSI generator (`build-windows-msi.ps1`) + updated CI/release callers
+ a committed, current `dist/windows/nono-machine.wxs` + a preserved
`dist/windows/nono-user.wxs` casing fix.
</objective>
