# Requirements — v2.4 Complete the Partial Ports + UPST4

**Defined:** 2026-05-12
**Core Value:** Close the partial-upstream-port debt accumulated in Phase 34, execute the v2.3 host-blocked carry-forwards, and absorb upstream v0.52.1 / v0.52.2 / v0.53.0 via UPST4 — leaving the fork in a state where future minor releases can be absorbed cluster-by-cluster without compounding D-20 manual-replay debt.

**Context:** v2.3 closed 2026-05-12 with documented carry-forwards. Phase 34's verification surfaced 10 NEEDS-FOLLOW-UP-PLAN deferrals (mostly partial upstream surface ports the fork only absorbed as much of as each phase needed); the v2.3 audit had 5 host-blocked requirements (RESL Unix backends + PKGS streaming/auto-pull) awaiting Linux/macOS host coverage. The Phase 33 ADR's "per upstream release, lazily-evaluated" cadence rule fires for v0.52.1+ (3 new tags landed post-audit cutoff).

**Scope shape:** 6-7 phases (Phases 35–40, plus optional 36.5 + 38). 14 requirements across 5 categories (PORT-CLOSURE, RESL-NIX, PKGS, UPST4, AAHX-HOST). Cross-platform-first by construction. Heaviest plan is Phase 36 deep-closure (4-6 weeks); shortest is Phase 35 quick wins (~2 weeks).

**Out of scope (explicit deferrals to v2.5 or later):**
- **v2.5-FU-1** — audit-bundle shim removal (Phase 27.2 deferral)
- **v2.5-FU-2** — cmd_verify v2 JSON schema (Phase 27.2 deferral)
- **AIPC G-04 wire-protocol compile-time tightening** — v3.0
- **WR-02 EDR HUMAN-UAT** — v3.0 (EDR-instrumented runner required)
- **P32-DEFER-005** (sigstore-verify 0.6.5 → 0.6.6) — v2.4 stretch or v2.5

## PORT-CLOSURE — Complete the Partial Upstream Ports (Phase 34 follow-ups)

Context: Phase 34 absorbed upstream v0.41–v0.52 cross-platform parity work but left 10 NEEDS-FOLLOW-UP-PLAN deferrals where each phase had to choose between cluster-shipping discipline (D-19 per-commit cherry-pick + D-34-E1 Windows-only-files invariant + D-34-D2 close-gate) and absorbing the full upstream surface those clusters assumed existed. v2.4 closes that debt.

### REQ-PORT-CLOSURE-01 — Windows execution-path env-filter wiring (P34-DEFER-08a-1)

- **What:** Wire `allowed_env_vars` / `denied_env_vars` consumption into `crates/nono-cli/src/exec_strategy_windows/` so Windows execution actually enforces the env filter Plan 34-08a added to the cross-platform path. Currently the fields are `#[cfg_attr(target_os = "windows", allow(dead_code))]` — forwarded cross-platform but no-op on Windows.
- **Enforcement:** Windows path. Mirrors the Unix `ExecConfig` env-filter consumption added in Plan 34-08a.
- **Security:** Closes a parity gap — operator-controlled `deny_vars` is currently Linux/macOS-only; Windows users invoking `nono run --env-deny SECRET_*` see no effect.
- **Acceptance:**
  1. `nono run --env-deny KEY -- <cmd>` on Windows strips `KEY` from the child env (verified via `nono run --env-deny KEY -- powershell -c '$env:KEY'`).
  2. `nono run --env-allow KEY1,KEY2 -- <cmd>` on Windows strips all env vars except KEY1, KEY2 (allowlist semantics).
  3. `780965d7` empty-allow fail-closed invariant holds on Windows: `allow_vars: []` + no `deny_vars` → DENY-ALL.
  4. New Windows-gated tests covering deny_vars precedence + empty-allow fail-closed.
- **Maps to:** Phase 34 deferred-items § P34-DEFER-08a-1.

### REQ-PORT-CLOSURE-02 — Full deprecated_schema module port (P34-DEFER-04b-1)

- **What:** Port upstream's 824-line `deprecated_schema` module verbatim (`LegacyPolicyPatch` rewriter, per-key `DeprecationCounter`, `--strict` mode for `nono profile validate`, alias inventory enforcement via `scripts/test-list-aliases.sh` + `scripts/lint-docs.sh`). Internal Rust identifier rename `override_deny` → `bypass_protection` across the 210-callsite surface. Canonical sections `groups`, `commands.{allow,deny}`, `filesystem.{deny,bypass_protection}` in `Profile` / `LoadedProfile` structs. JSON schema (`nono-profile.schema.json`) restructure. Built-in profile data migration. Embedded profile-authoring guide + `docs/cli/features/profiles-groups.mdx` + `docs/cli/usage/flags.mdx` migration.
- **Enforcement:** Cross-platform. Replaces the pragmatic Option C rename-acceptance (serde alias + clap visible_alias + AtomicBool one-time stderr deprecation warning) that Plan 34-04b landed.
- **Security:** Strict mode (`nono profile validate --strict`) gives operators a fail-closed lever for environments that must reject legacy keys.
- **Acceptance:**
  1. `LegacyPolicyPatch` struct present; legacy `override_deny` keys deserialize into it and rewrite to canonical `bypass_protection` post-parse.
  2. Per-key `DeprecationCounter` tracks first-encounter-per-process emission semantics (matches upstream).
  3. `nono profile validate --strict` fails closed on legacy keys (exit code non-zero + clear error pointing to the canonical key).
  4. JSON schema fixture restructured to upstream canonical form; `scripts/regenerate-schema.sh` produces matching output.
  5. Built-in profiles (claude-code, codex, opencode, claude-no-keychain) migrated to canonical sections.
  6. `docs/cli/usage/flags.mdx` + `docs/cli/features/profiles-groups.mdx` updated; `scripts/lint-docs.sh` alias-inventory check passes.
- **Maps to:** Phase 34 deferred-items § P34-DEFER-04b-1.

### REQ-PORT-CLOSURE-03 — Profile drafts feature absorption (P34-DEFER-04b-2)

- **What:** Absorb upstream `829c341a` profile drafts feature: `nono profile promote` subcommand, `--draft` flag, `crates/nono-cli/src/package_status.rs` module, `NonoError::ActionRequired` variant, profile-drafts directory infrastructure with atomic operations + base-hash verification.
- **Enforcement:** Cross-platform. New CLI surface; Windows path uses fork's `%LOCALAPPDATA%\nono\profile-drafts\` (mirroring package storage convention).
- **Security:** Atomic ops + base-hash verification prevent draft-corruption races + tampered-draft promotion. `ActionRequired` error variant gives operators a clear escalation path when a draft conflicts with an in-flight profile change.
- **Acceptance:**
  1. `nono profile draft <name>` creates a profile-drafts directory entry with an atomic write.
  2. `nono profile promote <name>` rewrites the canonical profile from the draft after base-hash verification.
  3. Base-hash mismatch produces `NonoError::ActionRequired` with a clear `expected: <hash>; actual: <hash>; resolve via: nono profile draft --refresh` message.
  4. Windows path uses `%LOCALAPPDATA%\nono\profile-drafts\` per Phase 22-03 PKG-02 storage convention.
  5. Round-trip serde test: draft → promote → re-deserialize as canonical Profile.
- **Maps to:** Phase 34 deferred-items § P34-DEFER-04b-2.

### REQ-PORT-CLOSURE-04 — yaml_merge wiring trio + base abstraction (P34-DEFER-06-1 + P34-DEFER-09-2)

- **What:** Port upstream `24d8b92a` `wiring.rs` abstraction base (~1761 LOC; idempotent JSON-merge install records) + the yaml_merge wiring trio (`242d4917` serde_yaml_ng pin, `802c8566` rustfmt, `d44f5541` yaml_merge directive). Without the base abstraction, fork users patching upstream profiles via YAML overlay can't use upstream's `yaml_merge` directive shape.
- **Enforcement:** Cross-platform. Composes with fork's existing `crates/nono-cli/src/wiring.rs` (currently a thin shim).
- **Security:** Idempotent install records prevent corrupted-state-on-replay attacks; `serde_yaml_ng` pin guards against the `serde_yaml` deprecation drift.
- **Acceptance:**
  1. `wiring.rs` abstraction landed with idempotent JSON-merge install records.
  2. `nono profile patch --yaml <overlay>` accepts `yaml_merge:` directives matching upstream semantics.
  3. `serde_yaml_ng` pinned to 0.10.0 in `crates/nono-cli/Cargo.toml`.
  4. Reversal failure test from upstream `242d4917` landed.
- **Maps to:** Phase 34 deferred-items § P34-DEFER-06-1 + § P34-DEFER-09-2.

### REQ-PORT-CLOSURE-05 — `b5f0a3ab` deep ExecConfig refactor + escape-quote pipeline (P34-DEFER-08b-1 + P34-DEFER-08b-2)

- **What:** Absorb upstream `b5f0a3ab` (11 files / +721 / -118; macos learn + run diagnostics + PTY-quiet-period + profile-save resilience) by refactoring the fork's ExecConfig to either accept upstream's shape or expose the equivalent diagnostics via fork's existing 8+ extra fields (capability_elevation, resource_limits, audit_signer, no_diagnostics, threading, protected_paths, profile_save_base, startup_timeout, allowed_env_vars, denied_env_vars, bypass_protection_paths). Then absorb `bbdf7b85` (escape-quote structured-property pipeline) which depends on the refactor.
- **Enforcement:** Cross-platform.
- **Security:** macOS learn/run diagnostic improvements close visibility gaps; PTY-quiet-period prevents leaked-input race; profile-save resilience is a non-repudiation hardening.
- **Acceptance:**
  1. `ExecConfig` accepts upstream's shape (or fork preserves its fields via a serde merge layer with explicit override semantics).
  2. macOS learn diagnostic improvements from `b5f0a3ab` land (verified via `nono learn` output deltas).
  3. PTY-quiet-period absorbed without regressing fork's Phase 17 attach-streaming or Phase 31 broker ConPTY path.
  4. `bbdf7b85` escape-quote structured-property test from upstream passes.
- **Maps to:** Phase 34 deferred-items § P34-DEFER-08b-1 + § P34-DEFER-08b-2.

### REQ-PORT-CLOSURE-07 — Windows test-harness hygiene (P34-DEFER-01-1 + P34-DEFER-10-1)

- **What:** Fix `query_ext::test_query_path_denied` Windows UNC long-path test flake (pre-existing carry-forward across all Phase 34 plans) + `policy show/diff --json` Rust Debug leak (`test_policy_show_json_no_rust_debug_syntax` + `test_policy_diff_json_no_rust_debug_syntax`).
- **Enforcement:** Windows path only.

- **Acceptance:**
  1. `cargo test query_ext::test_query_path_denied` passes deterministically on Windows.
  2. `nono profile show --json` and `nono profile diff --json` emit clean JSON (no `Debug` derive leak); regression tests pass.
