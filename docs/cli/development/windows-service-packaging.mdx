# Windows Service Packaging

This document describes the Windows service-and-driver packaging contract for `nono-wfp-service` and `nono-wfp-driver.sys` as shipped in the supported machine-MSI release path.

The machine-scope MSI (`nono-vX.Y.Z-x86_64-pc-windows-msvc-machine.msi`) bundles both binaries alongside `nono.exe` at `[InstallFolder]` (typically `C:\Program Files\nono\`). The user-scope MSI (`...-user.msi`) deliberately omits both — see [Windows Installer Strategy](/cli/development/windows-installer-strategy) for the full scope model, and the "Machine MSI only" section below for why.

## What is nono-wfp-service

`nono-wfp-service.exe` is the Windows Filtering Platform (WFP) backend service for nono.
It is a demand-start service: the SCM registers it at install time but does not start it
automatically at boot. nono starts it on demand when network blocking is requested.

The machine MSI installs the service binary alongside `nono.exe` in the nono install folder (typically `C:\Program Files\nono\`).

> **Note:** The current service runtime is a placeholder. The binary installs the WFP
> service contract and the first user-mode WFP primitives, but the full service backend is
> not yet complete. The service fails closed (exit code 3) when invoked in service mode
> before the real backend ships.

## Machine MSI only

The machine MSI ships both the user-mode service binary (`nono-wfp-service.exe`) and the pre-signed kernel driver (`nono-wfp-driver.sys`) at `[InstallFolder]`. Service registration follows a two-step model:

1. **User-mode service:** The MSI's `<ServiceInstall>` directive registers `nono-wfp-service` with the Service Control Manager (SCM) as demand-start under `LocalSystem` at install time. No CLI action is needed for this step — the MSI handles it.
2. **Kernel driver:** The MSI lands `nono-wfp-driver.sys` at `[InstallFolder]` but does not register it. WiX's `<ServiceInstall>` directive can only model user-mode services and cannot express `SERVICE_KERNEL_DRIVER`. The CLI command `nono setup --install-wfp-driver` completes registration post-install via `sc create ... type=kernel`. POC operators run this once from an elevated PowerShell prompt; the full post-install sequence is documented in the [Windows POC Handoff Cookbook § Post-install WFP registration](/cli/development/windows-poc-handoff#post-install-wfp-registration-machine-msi-only).

The machine MSI runs with admin elevation — the natural fit for both the `LocalSystem` service registration done by the MSI itself and the kernel-driver registration done by the CLI command.

The user-scoped MSI (`nono-vX.Y.Z-x86_64-pc-windows-msvc-user.msi`) deliberately omits **both** `nono-wfp-service.exe` and `nono-wfp-driver.sys`. The kernel driver cannot load from per-user `LocalAppData` and a `LocalSystem` service registration is not possible without admin elevation; shipping the binaries to user scope would produce an MSI that installs successfully but fails at runtime. The runtime probe in `exec_strategy_windows::network::probe_wfp_runtime` fail-closes with a directive message if WFP runtime activation is attempted from a user-scope install. Using the machine MSI is required for service-backed network enforcement.

See [Windows Installer Strategy](/cli/development/windows-installer-strategy) for the full
scope model.

## Service registration details

The MSI registers the service with the following Windows Service Control Manager (SCM)
attributes:

| Attribute     | Value                                    |
|---------------|------------------------------------------|
| Service name  | `nono-wfp-service`                       |
| Display name  | `nono WFP Service`                       |
| Binary path   | `[InstallFolder]\nono-wfp-service.exe --service-mode` |
| Account       | `LocalSystem`                            |
| Startup type  | Demand start (not automatic)             |
| Error control | Normal                                   |

## Verifying service registration after install

After installing the machine MSI, confirm the user-mode service is registered:

```powershell
Get-Service -Name nono-wfp-service
```

Expected output includes `Status` and `Name`. The service may be `Stopped` immediately after
install — demand-start services are registered but not automatically started.

To confirm the SCM entry in detail:

```powershell
sc.exe query nono-wfp-service
```

Expected fields include `SERVICE_NAME: nono-wfp-service` and `START_TYPE: 3 DEMAND_START`.

## Upgrade lifecycle

When upgrading to a newer machine MSI:

1. The MSI install sequence stops the running `nono-wfp-service` before replacing files.
2. The old service binary (`nono-wfp-service.exe`) is replaced with the new version.
3. The new MSI's install sequence re-registers and starts the service.

No manual service control is needed for a standard upgrade. After upgrade, confirm:

```powershell
sc.exe query nono-wfp-service
# Expected: STATE: RUNNING or STOPPED (demand-start, not auto-start)

(Get-Service -Name nono-wfp-service).Status
# Expected: Running or Stopped — service entry exists and is not a duplicate
```

There should be no duplicate or orphaned service entry after upgrade. If `sc.exe query`
returns an error, see Troubleshooting below.

## Uninstall lifecycle

When uninstalling the machine MSI through Apps & Features:

1. The MSI install sequence stops the running `nono-wfp-service`.
2. The MSI removes the service entry from SCM.
3. The service binary and all installed files are removed from the install folder.

After uninstall, confirm the service is gone:

```powershell
Get-Service -Name nono-wfp-service -ErrorAction SilentlyContinue
# Expected: no output (service entry does not exist)
```

If the service entry remains after uninstall, see Troubleshooting below.

## Manual service control

Use these commands to start, stop, or query the service manually for troubleshooting:

```powershell
# Query current state
sc.exe query nono-wfp-service

# Start the service (requires admin)
sc.exe start nono-wfp-service

# Stop the service (requires admin)
sc.exe stop nono-wfp-service
```

Or with PowerShell cmdlets:

```powershell
# Query
Get-Service -Name nono-wfp-service

# Start (requires admin)
Start-Service -Name nono-wfp-service

# Stop (requires admin)
Stop-Service -Name nono-wfp-service
```

## Troubleshooting

### Service entry remains after uninstall

If `Get-Service nono-wfp-service` returns an entry after MSI uninstall, the SCM entry can be
removed manually from an elevated prompt:

```powershell
sc.exe delete nono-wfp-service
```

This should only be needed if the MSI uninstall was interrupted or if a previous install
left an orphaned entry.

### Service fails to start

The current service binary fails closed (exit code 3) when the WFP driver artifact is
absent or when invoked before the full service runtime is implemented. This is the expected
behavior for the current placeholder runtime.

To inspect the failure:

```powershell
# Check Windows Event Log for service start failures
Get-EventLog -LogName System -Source "Service Control Manager" -Newest 20 |
    Where-Object { $_.Message -like "*nono*" }
```

### Duplicate service entry after upgrade

If an upgrade leaves duplicate entries (unusual with the MajorUpgrade-based MSI), stop and
remove the stale entry manually:

```powershell
sc.exe stop nono-wfp-service
sc.exe delete nono-wfp-service
```

Then re-run the upgrade MSI to re-register the service cleanly.

## Relationship to the RC checklist

Verify the service lifecycle in the [Windows Release Candidate Checklist](/cli/development/windows-release-candidate-checklist):

- **Packaging Gate** — machine MSI installs `nono-wfp-service.exe` and `nono-wfp-driver.sys` alongside `nono.exe`
- **Upgrade and Uninstall Gate** — after upgrade: no duplicate service entry; after
  uninstall: `Get-Service` returns no entry for `nono-wfp-service`

## What is not in scope

- **Automatic start at boot** — the service is demand-start. Auto-start will be reconsidered
  when the full WFP backend ships.
- **Automatic driver registration by the MSI** — the machine MSI bundles `nono-wfp-driver.sys` but does not register it as a kernel-mode service. Registration is a post-install step via `nono setup --install-wfp-driver` (see "Machine MSI only" above). A WiX `CustomAction` to invoke the CLI command at `InstallFinalize` may be added in a future release once the post-install path is empirically stable.
- **User MSI service registration** — the user MSI does not register the service.
  Per-user service install requires elevated permissions not compatible with a user install.
