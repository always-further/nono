title: ETI Composable Chaining Witness
description: See Ephemeral Tool Isolation enforce caller-aware tool chains with a coding-agent-shaped workflow
---

This witness is meant to make ETI feel concrete. It uses `sh` as a small
stand-in for an agent command that may use `git`, while direct `git` and direct
executable-path bypasses are denied. Optionally, it extends the chain to
`sh -> git -> ssh` so you can see a private-dependency-shaped workflow where the
SSH agent is only reachable at the final hop.

<Note>
  ETI v1 is Linux-only. Run these checks on a Linux host with Landlock support.
</Note>

## What This Proves

Static command policy has to answer coarse questions like "may this session use
`git`?" or "may this session use `ssh`?" ETI answers a more useful question:

> Who is calling the tool, and does that caller relationship justify the risky
> capability?

In the witness profile:

- the outer session may run `sh`;
- `sh` represents a coding agent or agent task runner;
- `sh` may call `git`;
- direct session `git` is denied;
- direct `/usr/bin/git` is denied, so the shim cannot be bypassed;
- in the optional SSH profile, `git` may call `ssh`;
- direct session `ssh` and direct `/usr/bin/ssh` are denied;
- `SSH_AUTH_SOCK` is exposed only to `ssh.from.git`.

That is the composability ETI adds: the same command has different authority
depending on the caller chain.

ETI also narrows executable launch. The outer process gets the initial program
and ETI shims; helper tools are expected to go through explicit command-policy
hops rather than inheriting the whole host `PATH`.

## Quick Witness

Build the CLI:

```bash
make build-cli
```

Run the witness script:

```bash
NONO=./target/debug/nono scripts/eti-composable-witness.sh
```

Expected results:

- profile validation succeeds;
- `sh -> git` succeeds;
- direct `git --version` is denied;
- direct canonical `$(command -p -v git) --version` is denied.

This first pass does not need network access, SSH, or credentials. It proves the
caller-aware command boundary locally.

## Optional SSH Chain

To witness a real private-dependency-shaped chain, prepare an SSH agent and a
known host entry:

```bash
ssh-add -l
ssh-keygen -F github.com -f ~/.ssh/known_hosts
```

If `github.com` is not in `known_hosts`, prime it outside nono:

```bash
ssh -o IdentityFile=none -T git@github.com
```

Then run:

```bash
ETI_SSH_PROBE=1 \
ETI_SSH_REMOTE=git@github.com:OWNER/PRIVATE-REPO.git \
NONO=./target/debug/nono \
scripts/eti-composable-witness.sh
```

Expected results:

- `sh -> git -> ssh` succeeds for a repo your SSH key can read;
- direct `ssh -V` through `PATH` is denied;
- direct canonical `/usr/bin/ssh -V` is denied;
- the SSH agent socket is not available to the outer session or to `git`; it is
  injected only into the selected `ssh.from.git` child policy.

If you use a public GitHub SSH remote and your key is not registered with
GitHub, the remote may return `Permission denied (publickey)`. That means SSH
reached GitHub but authentication failed outside nono. Use a repo and key pair
that already work outside nono when you want the full success path.

## Why This Matters For Agents

The same pattern maps onto real coding agents:

```text
session -> codex -> git -> ssh
session -> claude -> git -> ssh
session -> npm -> git -> ssh
session -> terraform -> aws
```

Without chaining, granting `ssh` globally exposes an ambient identity: the agent
can use your SSH agent for direct shell access, arbitrary Git remotes, `scp`, or
port forwarding. Denying `ssh` globally avoids that exposure but breaks common
developer workflows such as private dependencies and submodules.

With ETI, the policy can say:

```text
codex may call git
git may call ssh
ssh.from.git gets SSH_AUTH_SOCK and TCP/22
ssh.from.session is denied
```

The policy is not trusting `ssh` in the abstract. It is trusting `ssh` only in
the specific caller context that explains why SSH is needed.

## Manual Profile Shape

The witness script generates temporary profiles dynamically so it can pin the
actual `sh`, `git`, and `ssh` binaries on your machine. The core shape is:

```json
{
  "extends": "default",
  "workdir": { "access": "none" },
  "command_policies": {
    "session_can_use": ["sh"],
    "commands": {
      "sh": {
        "executable": "/usr/bin/sh",
        "can_use": ["git"],
        "sandbox": {
          "fs_read": ["."],
          "fs_write": ["."]
        }
      },
      "git": {
        "from": {
          "sh": {
            "fs_read": ["."],
            "fs_write": ["."]
          },
          "session": "deny"
        }
      }
    }
  }
}
```

The SSH extension adds:

```json
{
  "command_policies": {
    "credentials": {
      "ssh-agent": {
        "type": "ssh-agent",
        "socket": "$SSH_AUTH_SOCK"
      }
    },
    "commands": {
      "git": {
        "can_use": ["ssh"]
      },
      "ssh": {
        "from": {
          "git": {
            "use_credentials": ["ssh-agent"],
            "argv_prepend": ["-o", "IdentityFile=none"],
            "fs_read_file": ["$HOME/.ssh/known_hosts"],
            "network": {
              "tcp_connect_ports": [22]
            }
          },
          "session": "deny"
        }
      }
    }
  }
}
```

## Turning This Into A Codex Or Claude Profile

The witness uses `sh` because it is small and its execution dependencies are
covered by ETI's generic ELF baseline. For a real agent profile, replace `sh`
with the agent command name and give that command the filesystem grants it needs:

```text
session_can_use: [codex]
codex.can_use: [git]
git.from.codex: read/write project, sanitized git env
git.can_use: [ssh]
ssh.from.git: ssh-agent, known_hosts, TCP/22
ssh.from.session: deny
```

The important part is not the specific first command. The important part is that
the credential-bearing command is only reachable through the workflow that
justifies it.

## Audit Checks

After a run:

```bash
nono audit list
nono audit show <SESSION_ID> --json | jq '.command_policy_events'
```

Useful evidence to look for:

- direct `git` denial has caller kind `session`;
- direct canonical `git` denial records direct-exec bypass denial;
- chained `git` launch records caller command `sh`;
- optional chained `ssh` launch records caller command `git`;
- denied `ssh` from the session records `session_can_use missing` or explicit
  session denial.
